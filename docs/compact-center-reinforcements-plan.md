# 中心凝集・段階的増員 実装プラン

## 現状と変更後

- 現在の味方は方向転換時に中心へ戻る傾向があるが、中心から0.05以上離れた場合の80%バイアスに留まり、衝突で生じた前後・左右の押し出し量は残り続ける。変更後は中心回帰バイアスを強め、衝突変位にも継続的な中心復元力を加える。
- 現在の隊列は行ごとに前後0.027の間隔を取り、衝突による `pushY` が減衰しないため、人数増加時に前後へ細長くなりやすい。変更後は基礎行間隔を狭め、特に前後方向の `pushY` を速く中心へ戻して、横幅と奥行きの釣り合った密な集団を維持する。
- 現在の正ゲートは目標人数まで全員を同じフレームで追加し、通常隊列の末尾位置へ配置する。変更後は増員数を予約し、短い一定間隔で1体ずつ集団中心へ生成する。各生成直後に味方同士の押し合いを解決し、直前に中心へ出た個体を次の個体が外側へ押し出す。

## スコープ

### 対象

- 味方の中心回帰確率・開始距離と、衝突変位の中心復元
- 隊列の前後間隔と、前後方向の広がり上限を決定する状態ロジック
- ゲートによる増員予約、一定間隔の中心生成、生成ごとの押し合い
- 保留中増員を含む次のゲート計算、減員時の予約取り消し
- 決定的テスト、README更新、デスクトップ/モバイルのブラウザ検証

### 対象外

- 敵編成、敵AI、射撃、ダメージ、ゲートの正負・左右抽選
- 初期12体の段階生成、リスタート演出
- 棒人間の新しい描画アニメーションやパーティクル
- 一般的な物理エンジン、質量、速度、反発係数

## 状態モデルと描画

- `GameState.pendingAllyAdds` と `GameState.allySpawnCooldown` を追加する。HUDの人数は画面上に存在する `allies.length` のままとし、生成された時点で1人ずつ増える表示にする。
- ゲート通過時、正の結果が現在の実体数と予約数の合計を上回る場合は差分を予約する。負の結果では予約を取り消し、必要な実体をID順で減らす。
- `advanceAllyReinforcements` を追加し、0.12秒ごとに新規味方を1体生成する。新規個体の `formationX/Y` を `pushX/Y` で相殺して `playerX, PLAYER_Y` に置き、生成ごとに `resolveAllyContacts` を呼ぶ。
- 方向転換時の中心回帰を、中心から0.025以上で95%のバイアスへ強める。
- 基礎行間隔を0.020へ縮める。毎tick、`pushX` は緩やかに、`pushY` はより速く0へ近づけ、横方向の個体感を残しながら前後の長い尾を抑える。
- 描画は既存の味方配列とHUDをそのまま使うため新要素を追加しない。段階生成は実体配列の増加として自然に描画される。

## 決定的テストとブラウザ検証

- 20体および大人数の基礎隊列で、前後範囲が従来より短く、過度に細長くならない。
- 中心から離れた個体の少なくとも95%が内向き方向を選ぶ。
- `pushY` が経過時間ごとに0へ近づき、横の `pushX` より速く復元する。
- 増員予約直後は人数が増えず、最初のtickで中心に1体生成される。
- 0.12秒未満では次の1体が増えず、間隔到達ごとに1体ずつ増え、予約が0になる。
- 新規個体が中心に生成され、連続生成後は前の中心個体に押し出し変位が付く。
- 保留中の増員を含めて次のゲート演算が適用され、減員では予約が正しく取り消される。
- 既存の即時セットアップ用 `setSquadSize`、戦闘、落下、中ボス、ボス、リスタートテストが退行しない。
- `npm test` と `git diff --check` を実行する。
- ブラウザテストハーネス、1280x720、390x844、コンソール、横スクロール、Restartを検証する。
- 実ゲームで正ゲート通過後のHUDが段階的に増え、味方が中心から押し広がりつつ前後に伸びすぎないことを確認する。

## 実装手順とコミット境界

1. 本計画を追加し、ドラフトPRを作成する。コミット: `Plan compact center reinforcements`
2. 凝集力、短い前後隊列、増員予約・中心生成を `src/state.js` に実装し、決定的テストを追加する。コミット: `Add compact staggered reinforcements`
3. READMEを更新し、必要な描画調整を行う。コミット: `Document center-spawned allies`
4. 全差分レビューと自動・ブラウザ検証を行い、発見事項は独立コミットで解消する。

## 受け入れ条件

- 味方が従来より積極的に中心へ集まり、前後方向へ長い列を作りにくい。
- 正ゲート通過後、味方が中心から1体ずつ時間差で現れ、既存・直前生成の味方を押し広げる。
- HUD人数が生成の進行に合わせて段階的に増える。
- 連続ゲートや減員でも予約数と実体数が矛盾せず、最大人数を超えない。
- 既存ゲームルールが退行せず、自動テストとブラウザ検証が成功する。

## リスクと前提

- 「中心部分」はプレイヤー位置 `playerX` と基準Y `PLAYER_Y` の交点と解釈する。
- 0.12秒間隔は複数人が増える様子を視認でき、ゲーム進行を不必要に止めない値として採用する。
- 強い凝集と衝突分離は互いに逆向きに働く。固定の復元速度と既存の4回衝突パスで計算量と挙動を決定的に保つ。
- 初期配置とテスト用の直接人数設定は即時のまま維持し、実ゲームのゲート増員だけを段階化する。
