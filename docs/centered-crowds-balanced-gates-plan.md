# 集団集中・敵大編成・左右ゲート均衡 実装プラン

## 現状と変更後

- 現在の味方は個体ごとに左右へランダム移動し、方向転換時も隊列中心との位置関係を考慮しない。変更後は個体差を残しつつ、隊列中心から離れた味方ほど中心方向を選びやすくし、集団がまとまる傾向を持たせる。
- 現在の通常敵ウェーブは初期8体、最大18体を最大6列で生成する。変更後は初期12体、ステージごとに増加して最大30体とし、最大8列の密な複数行編成でまとまって出現させる。
- 現在のゲートは左側が高確率で加算または乗算、右側だけ減算を含む別々の抽選規則であり、左に居続ける戦略が有利である。変更後は正と負の選択肢を同じ確率で左右へシャッフルし、左右それぞれが正・負になる頻度を均等にする。

## スコープ

### 対象

- 味方のランダム移動方向を決める状態ロジックと、中心回帰傾向の決定的テスト
- 通常敵・archerウェーブの人数、列数、間隔を保った密な整列
- ゲートの正負生成と左右配置のランダム化
- READMEのゲームルール更新、自動テスト、デスクトップ/モバイルのブラウザ検証

### 対象外

- プレイヤー操作速度、味方同士の押し合い判定、射撃・ダメージ・HPの変更
- 中ボスと最終ボスの出現条件、サイズ、耐久力
- 敵ウェーブ生成間隔、敵AI、敵同士の衝突判定
- ゲート演算そのものや、ゲート通過判定の変更

## 状態モデルと描画

- `wanderProfile` に味方の現在の隊列中心からの横ずれを渡し、中心から一定距離以上では中心方向を高確率で、外側へは低確率で選ぶ。中心付近では既存の決定的な左右分散を維持する。
- 通常敵数を `10 + stage * 2`（12〜30体）へ増やし、最大8列、狭めた横・縦間隔で整列させる。既存の敵エンティティ構造、描画、独立追跡は維持する。
- ゲートペアは正演算（`+` または `x`）を1つ、負演算（`-` または `/`）を1つ生成し、追加の乱数で左右を交換する。各位置の正負確率を50%にする。
- 新しい描画要素は追加しない。既存描画が30体の敵、密な味方、左右に入れ替わる正負ラベルをキャンバス内に表示できることをブラウザで確認する。

## 決定的テストとブラウザ検証

- 中心から左へ離れた味方は右向き、右へ離れた味方は左向きの移動傾向を示し、中心までの距離が縮む。
- 中心付近の味方はID由来の決定的な個体差を維持する。
- 初期通常ウェーブが12体、後半ウェーブが最大30体になり、最大8列の複数行編成で全員がトラック内に収まる。
- 同じ乱数列ではゲート生成を再現でき、左右どちらにも正・負の選択肢が配置されるケースを検証する。
- 各ゲートペアが正と負を1つずつ含み、既存の全演算が適用可能であることを確認する。
- `npm test` と `git diff --check` を実行する。
- ブラウザのテストハーネスが全件成功し、コンソールエラーがないことを確認する。
- 1280x720で集団の中心回帰、12体以上の敵整列ウェーブ、左右ゲートの入れ替わり、Restartを確認する。
- 390x844で多数の敵とゲートがキャンバス内に収まり、横スクロールがないことを確認する。

## 実装手順とコミット境界

1. 本計画を追加し、ドラフトPRを作成する。コミット: `Plan centered crowds and balanced gates`
2. 味方の中心回帰、敵大編成、左右均衡ゲートを `src/state.js` に実装し、`src/tests.js` に決定的テストを追加する。コミット: `Tune crowd formations and gate balance`
3. READMEを更新し、表示上必要な微調整を行う。コミット: `Document denser balanced gameplay`
4. 全差分レビューと自動・ブラウザ検証を実施し、発見事項があれば内容ごとに独立コミットで修正する。

## 受け入れ条件

- 味方がばらけ続けず、個別に動きながらも隊列中心へ戻る傾向が視認できる。
- 通常敵が開始時から12体以上、密な横列・複数行の集団として出現する。
- 左右どちらか一方に居続けるだけでは正のゲートを選び続けられず、正負の配置が左右で均等に入れ替わる。
- 既存の操作、戦闘、押し合い、端からの落下、中ボス、ボス、リスタートが退行しない。
- 自動テストと指定ブラウザ検証が成功する。

## リスクと前提

- 「集団の中心」はプレイヤー位置を基準とする隊列中心と解釈し、全味方座標の毎フレーム平均を新たに状態保存しない。
- 中心回帰を強制移動にすると個体感を失うため、方向転換時の重み付けとして実装する。
- 敵数30体は描画・標的探索コストを増やす。上限を固定し、デスクトップとモバイルで動作を確認する。
- 正負ゲートは各ペアに1つずつ配置する。難易度の急変を避けるため、値の既存範囲は維持する。
